**작성일**: 2026-04-05  
**대상**: 프로젝트에 처음 합류하는 팀원, 또는 trust 로직을 처음 접하는 분  
**읽는 데 걸리는 시간**: 약 10분

---

## 먼저 읽어야 할 것 — 이 시스템이 "무엇을 하는가"

> **이 시스템은 기사의 사실 여부를 판단하지 않습니다.**

이 점이 가장 중요합니다. 시작 전에 확실히 구분해두겠습니다.

| 구분 | 설명 | 우리 시스템 |
|------|------|------------|
| 팩트체크 | "이 기사의 내용이 사실인가?" | ❌ 하지 않음 |
| 패턴 경고 | "이 기사가 가짜뉴스의 형식적 특징을 보이는가?" | ✅ 이것을 함 |

연합뉴스 기자가 정성껏 쓴 기사도 수치가 틀릴 수 있고,
무명 블로거의 글이 사실일 수도 있습니다.
우리 시스템은 **"어떻게 썼는가"** 를 봅니다. **"무엇이 맞는가"** 는 보지 않습니다.

이 전제를 이해하고 나서 아래를 읽으면, 모든 설계 결정이 자연스럽게 납득됩니다.

---

## 1. 왜 이 시스템이 필요한가

뉴스 기사를 읽을 때 "이 기사, 좀 이상한데?" 하는 감이 올 때가 있습니다.

- 제목이 너무 자극적이다
- 전문가 의견 없이 기자 혼자 주장한다
- 앞에서 한 말과 결론이 다르다
- 언론사를 들어보지 못했다

이 '감'을 **5개의 구체적인 기준**으로 수치화한 것이 trust.py입니다.  
사용자가 기사 하나하나를 직접 검증할 수 없으니, 시스템이 먼저 패턴을 걸러주는 역할입니다.

---

## 2. 5가지 평가 기준

기사 하나를 받으면 다음 5개 항목으로 분석합니다.

### ① 출처 신뢰성 (source_credibility) — 가중치 20%

**"이 기사가 어떤 언론사에서 나왔는가?"**

AI가 아닌 **미리 만든 등급 사전**으로 판단합니다.

```
10점 — 연합뉴스, KBS, MBC, SBS, YTN
 8점 — 조선일보, 중앙일보, 한겨레, 경향신문 등 주요 일간지
 6점 — 머니투데이, 뉴시스, 이데일리 등 경제·전문지
 3점 — SNS, 블로그, 유튜브 등
 5점 — 등록되지 않은 언론사 (중립 처리)
```

LLM을 쓰지 않는 이유: 출처 등급은 주관적 판단이 아닌, 고정된 사실이기 때문입니다.  
AI에게 물어볼 필요가 없고, 물어보면 오히려 일관성이 깨집니다.

---

### ② 근거 지지도 (evidence_support) — 가중치 25%

**"이 기사에 주장을 뒷받침하는 근거가 있는가?"**

LLM에게 3가지를 묻습니다:
- 구체적인 수치나 통계가 포함되어 있는가?
- 실명의 전문가 또는 공식 기관이 인용되어 있는가?
- 인용된 자료의 출처가 명시되어 있는가?

근거 없이 주장만 늘어놓는 기사 → 가짜뉴스의 가장 흔한 패턴.  
그래서 5개 기준 중 **가중치가 가장 높습니다 (25%)**.

---

### ③ 문체 중립성 (style_neutrality) — 가중치 25%

**"이 기사가 감정을 자극하거나 편향된 언어를 쓰고 있는가?"**

LLM에게 3가지를 묻습니다:
- 기사 제목에 감정적이거나 자극적인 표현이 없는가?
- 본문에 선동적이거나 편향된 언어가 사용되지 않았는가?
- 사실과 의견이 명확히 구분되어 있는가?

"[충격]", "[단독]", "결국...", "어이없게도..." 같은 표현들이 이 항목을 낮춥니다.  
감정적 글쓰기는 신뢰도를 가장 직관적으로 떨어뜨리는 신호입니다.

---

### ④ 논리 일관성 (logical_consistency) — 가중치 20%

**"이 기사가 처음부터 끝까지 같은 이야기를 하고 있는가?"**

LLM에게 3가지를 묻습니다:
- 앞부분 주장과 결론이 일치하는가?
  *(앞에서 A라고 했는데 결론에서 B라고 하면 no)*
- 문단 간 서로 모순되는 문장이 없는가?
  *(한 문단에서 긍정했다가 다른 문단에서 부정하면 no)*
- 제목의 주장이 본문으로 실질적으로 뒷받침되는가?

처음에 강한 주장을 하고 본문은 엉뚱한 이야기로 채우는 기사 → 이 항목에서 걸립니다.

---

### ⑤ 어뷰징 위험도 (clickbait_risk) — 가중치 -10%

**"이 기사가 클릭을 유도하기 위해 내용을 과장하거나 왜곡하고 있는가?"**

LLM에게 3가지를 묻습니다:
- 제목이 본문 내용을 과장하거나 왜곡하고 있는가?
- 독자의 클릭을 유도하기 위한 자극적 표현이 제목에 있는가?
- 기사 내용이 제목이 암시하는 것과 실질적으로 다른가?

> **주의: 이 기준은 점수 계산 방식이 다릅니다.**  
> 낚시성이 높을수록(yes가 많을수록) 점수가 **올라가는** 방향으로 LLM이 답하고,  
> 가중치가 **음수(-10%)** 이기 때문에 최종 점수에서 **감산**됩니다.  
> 낚시성 기사 = 전체 신뢰도 점수가 낮아지는 방식입니다.

## 3. 분석 과정 — 기사 하나가 점수가 되는 흐름

```
기사 본문 입력
    │
    ▼
[전처리] 텍스트가 너무 길면 앞 2500자 + 뒤 1500자만 사용 (최대 4000자)
    │      이유: 기사 앞부분(핵심 주장)과 뒷부분(결론)을 모두 보기 위함
    │
    ├──────────────────────────────────────────────────────────────
    │                                                              │
    ▼                                                             ▼
[룰 기반]                                              [LLM 분석]
출처명을 등급 사전에서 조회                          Gemini에게 12개 질문 (1회 호출)
→ source_credibility 점수 (0~10)                    → 각 질문에 yes/no/uncertain 반환
    │                                                              │
    └──────────────────────┬───────────────────────────────────────┘
                           │
                           ▼
              [yes/no → 점수 변환]
              yes = 1.0 / no = 0.0 / uncertain = 0.3
              3문항 평균 × 10 → 각 기준 0~10점
              (uncertain을 0.5가 아닌 0.3으로 처리하는 이유:
               LLM이 "모르겠음"으로 회피할 때 중립보다 약간 부정적으로 간주)
                           │
                           ▼
              [교차 패널티 적용]
              문체가 감정적(style < 4점) + 근거가 없음(evidence < 4점)
              → 동시에 해당하면 raw 점수 -10점
              이유: 이 조합은 가짜뉴스의 가장 전형적인 패턴이기 때문
                           │
                           ▼
              [가중 합산 → 0~100점]
              출처(20%) + 근거(25%) + 문체(25%) + 논리(20%) - 낚시(10%)
                           │
                           ▼
              [최종 판정]
              70점 이상 → "패턴 정상"      (likely_true)
              40~69점   → "일부 주의 필요" (uncertain)
              40점 미만 → "가짜뉴스 패턴 감지" (likely_false)
              분석 실패  → "분석 실패"     (unanalyzed)
```

## 4. 실제 출력 예시

기사 하나를 분석하면 다음과 같은 결과가 나옵니다.

```json
{
  "score": 72,
  "verdict": "likely_true",
  "reason": "전문가 인용과 통계가 포함되어 있으며 중립적 문체로 작성되었습니다. 제목과 본문 내용이 일치합니다.",
  "per_criteria": {
    "source_credibility":  { "score": 8,  "reason": "등록 언론사: 중앙일보" },
    "evidence_support":    { "score": 7,  "reason": "통계청 자료 인용 / 경제학 교수 실명 인용" },
    "style_neutrality":    { "score": 8,  "reason": "중립적 서술 / 사실-의견 구분 명확" },
    "logical_consistency": { "score": 6,  "reason": "전반적으로 일관성 있음" },
    "clickbait_risk":      { "score": 2,  "reason": "제목이 본문과 일치함 / 자극적 표현 없음" }
  },
  "flags": {
    "short_article": false
  }
}
```

**UI에서는 이렇게 보입니다:**

```
패턴 분석 결과                    패턴 정상 · 72점  ⓘ
████████████████████████████░░░░

출처 신뢰성   ✅  8/10  중앙일보 등록 언론사
근거 지지도   ✅  7/10  통계청 자료 인용
문체 중립성   ✅  8/10  중립적 서술
논리 일관성   ⚠️  6/10  전반적으로 일관성 있음
어뷰징 위험도 ✅  2/10  제목과 본문 일치

이 분석은 기사의 작성 패턴을 평가한 것으로, 사실 여부를 보장하지 않습니다.
```

## 5. 설계 과정에서 해결한 주요 문제들

### 문제 1 — 어뷰징 위험도 점수가 반대로 작동했다

초기 설계에서 낚시성 기사가 오히려 **높은 점수**를 받는 버그가 있었습니다.

```
[버그 발생 원인]
1단계: LLM이 "낚시성 있음" → yes → 점수가 높아짐
2단계: "어뷰징은 역방향이다" → (1 - 점수) 로 뒤집음 → 점수가 낮아짐
3단계: 가중치 -0.10(음수) → 한 번 더 뒤집음 → 결국 낚시성 있으면 점수가 높아짐

반전이 두 번 일어나서 원래대로 돌아온 것
```

**해결**: 2단계(수동 반전)를 제거하고, 3단계(음수 가중치)만 남겼습니다.  
낚시성 있음 → yes → 점수 높음 → 음수 가중치로 감산 → 최종 점수 낮아짐.

---

### 문제 2 — 모든 기사 점수가 40~70점 사이에 몰렸다

LLM이 명확한 판단을 피하고 "uncertain"을 자주 반환하면서,  
대부분의 기사가 "일부 주의 필요(uncertain)" 구간에 몰리는 현상이 발생했습니다.

**해결**: uncertain 처리값을 0.5(중립)에서 **0.3(약간 부정)**으로 낮췄습니다.  
"모르겠다"는 답은 "좋다"가 아니라, 불확실한 것이므로 약간 불리하게 처리합니다.

---

### 문제 3 — 논리 모순 탐지가 작동하지 않았다

앞에서 A라고 하고 결론에서 B라고 해도 LLM이 "일관성 있음"으로 답하는 경우가 있었습니다.

**해결**: 프롬프트에 구체적인 판단 기준을 명시했습니다.

```
변경 전: "기사 내용에 모순이 없는가?"

변경 후: "기사 앞부분의 주장과 뒷부분 결론이 서로 일치하는가?
          (앞에서 A라 했는데 결론에서 B라 하면 no로 답하라)
          기사 내에 서로 모순되는 문장이 없는가?
          (한 문단에서 긍정했다가 다른 문단에서 부정하면 no로 답하라)"
```

## 6. 검증 — 얼마나 잘 작동하는가

**불변 조건 쌍 테스트 (invariant pairs)**

"이런 기사는 저런 기사보다 반드시 점수가 높아야 한다"는 9쌍의 기준을 만들어  
실제로 그렇게 작동하는지 확인했습니다.

| 쌍 | 내용 | 결과 |
|----|------|------|
| P1 | 연합뉴스 기사 > 출처 없는 기사 | ✅ PASS |
| P2 | 통계·전문가 인용 기사 > 주관적 주장 기사 | ✅ PASS |
| P3 | 중립 문체 > 선동적 문체 | ✅ PASS |
| P4 | 정상 제목 기사 > 낚시성 제목 기사 | ✅ PASS |
| P5 | 일관된 기사 > 결론이 모순된 기사 | ✅ PASS |
| P6 | 좋은 기사 > 모든 기준 나쁜 기사 | ✅ PASS |
| P7 | 출처·근거·문체 모두 좋은 기사 > 출처만 좋은 기사 | ✅ PASS |
| P8 | 근거 추가된 기사 > 근거 없는 기사 | ❌ FAIL |
| P9 | 단신 속보 처리 기대치 | ✅ PASS |

**9쌍 중 8쌍 통과 (88.9%)**

P8이 실패하는 이유: 출처 점수가 이미 8점(경향신문)으로 높아서,  
근거 추가 여부와 무관하게 출처 점수가 전체를 끌어올리는 구조적 문제입니다.  
현재로서는 수용하고 있습니다.

## 7. 알려진 한계 (팀원이 꼭 알아야 할 것)

1. **잘 쓰인 오보는 높은 점수를 받습니다.**  
   연합뉴스 형식으로 쓰였지만 내용이 틀린 기사 → 우리 시스템은 탐지 못합니다.  
   이건 설계 범위 밖의 문제입니다.

2. **단신 기사는 구조적으로 불리합니다.**  
   짧은 속보는 전문가 인용이나 통계를 넣기 어렵습니다.  
   → UI에서 단신 기사임을 안내 문구로 표시합니다.

3. **API 장애 시 "분석 실패(unanalyzed)"로 처리됩니다.**  
   이전에는 장애 시 0점/가짜뉴스로 처리해서 멀쩡한 기사가 오해받는 문제가 있었습니다.  
   지금은 명시적으로 "분석 실패" 상태를 따로 표시합니다.

4. **시스템 점수를 절대적 기준으로 사용하지 마세요.**  
   UI의 면책 문구("이 분석은 작성 패턴을 평가한 것입니다")가 핵심입니다.
