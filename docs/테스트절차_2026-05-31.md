# 테스트절차_2026-05-31

## 기준

- 대상: ET Dashboard / Explainable Trust 현재 프로젝트
- 목적: 수정 요구사항을 구현 가능한 품질 게이트로 검증한다.
- 원칙: 각 수정은 자동 테스트 1개 이상 또는 Playwright 수동 검증 1개 이상을 가진다.

## 1. 저장소 기준선 테스트

- 절차: `git fetch origin`, `git status --short --branch`, `git diff --stat`, `git diff --stat HEAD..origin/main`을 실행한다.
- 통과 기준: 로컬 변경, 원격 차이, 중첩 repo 상태가 문서화되고 의도하지 않은 삭제가 없어야 한다.
- 관련 수정: 로컬/GitHub 기준선 정리, 중첩 Git 저장소 처리.

## 2. 프론트엔드 빌드 테스트

- 절차: `cd frontend`, `npm run build`를 실행한다.
- 통과 기준: TypeScript 빌드와 Vite production build가 0 error로 끝난다.
- 관련 수정: 모든 프론트엔드 변경의 기본 회귀 검증.

## 3. 프론트엔드 린트 테스트

- 절차: `cd frontend`, `npm run lint`를 실행한다.
- 통과 기준: `.vite`가 검사 대상에서 제외되고 source 기준 0 error로 끝난다.
- 관련 수정: generated 파일 추적 제거, lint baseline 복구, API 타입 계약 강화.

## 4. 백엔드 컴파일 테스트

- 절차: 프로젝트 루트에서 `python -m compileall backend`를 실행한다.
- 통과 기준: 모든 backend Python 파일이 syntax error 없이 컴파일된다.
- 관련 수정: 백엔드 구조 변경, repo 계층 분리, admin/status 작업 변경.

## 5. 백엔드 import 테스트

- 절차: `python -c "from backend.main import app; print(app.title, len(app.routes))"`를 실행한다.
- 통과 기준: `psycopg2` 등 의존성 오류 없이 FastAPI app이 로드된다.
- 관련 수정: 백엔드 DB 의존성 일관화, startup 환경 검증.

## 6. API smoke 테스트

- 절차: TestClient 또는 실행 서버로 `/`, `/health`, `/api/articles`, `/api/admin/stats`를 호출한다.
- 통과 기준: 공개 API는 정상 응답, 관리자 API는 비밀번호 없을 때 401을 반환한다.
- 관련 수정: 백엔드 smoke test 추가, CORS와 환경변수 검증.

## 7. DB 실패 테스트

- 절차: DB 연결 실패 mock 상태에서 `/health`와 목록 API를 호출한다.
- 통과 기준: `/health`는 503, 목록 API는 명확한 오류 또는 안전한 empty response를 반환한다.
- 관련 수정: 백엔드 안정성, 외부 API 실패 처리 표준화.

## 8. 관리자 인증 UI 테스트

- 절차: Playwright로 `/admin` 진입, 잘못된 비밀번호, 올바른 비밀번호, 로그아웃을 수행한다.
- 통과 기준: 실패 메시지, 재입력, 인증 후 화면, 로그아웃 후 차단이 모두 정상 동작한다.
- 관련 수정: 관리자 인증 UX 정리, 관리자 비밀번호 저장 방식 개선.

## 9. 관리자 작업 상태 UI 테스트

- 절차: crawl/analyze/dedupe/keywords API를 success, pending, failure mock으로 전환해 화면을 확인한다.
- 통과 기준: 시작 범위, 진행 단계, 완료 수, 실패 원인, 재시도 행동이 구분되어 표시된다.
- 관련 수정: 관리자 작업 실행 UX 개선, 작업 상태 신뢰성 개선.

## 10. 메인 목록 UI 테스트

- 절차: `/`에서 정상 목록, empty 목록, API 실패, 긴 제목, 카테고리 필터를 검증한다.
- 통과 기준: 레이아웃 깨짐 없이 목록/빈 상태/오류/필터 초기화가 명확히 보인다.
- 관련 수정: 빈 상태와 오류 상태 보강, 모바일 레이아웃 검증.

## 11. 상세/신뢰도 UI 테스트

- 절차: `/article/:id`에서 trust 정상값, 0점, 누락값, 긴 설명, 관련 기사 empty를 검증한다.
- 통과 기준: 점수, 판정, 기준별 근거, 원문 링크, 관련 기사 상태가 분리되어 표시된다.
- 관련 수정: 신뢰도 설명 가독성 개선, API 타입 계약 강화.

## 12. 검색 품질 테스트

- 절차: 고정 기사 fixture로 keyword, semantic, hybrid query를 실행한다.
- 통과 기준: top result, 중복 제거, empty query, no result 케이스가 기대값과 일치한다.
- 관련 수정: 검색 결과 신뢰성 기준 추가, 기사 중복 제거 기준 명확화.

## 13. 키워드 데이터 테스트

- 절차: keywords가 null, 빈 문자열, JSON 문자열, 배열, 과다 배열인 fixture를 처리한다.
- 통과 기준: 저장 전 배열로 정규화되거나 명확한 실패로 분류된다.
- 관련 수정: 키워드 추출 데이터 품질 개선.

## 14. 반응형 화면 테스트

- 절차: Playwright로 375x812, 768x1024, 1440x900에서 `/`, `/article/:id`, `/admin`, `/feedback`를 캡처한다.
- 통과 기준: 텍스트 겹침, 가로 스크롤, 버튼 잘림, 표 overflow가 없어야 한다.
- 관련 수정: 모바일 레이아웃 검증, 관리자 작업 실행 UX 개선.

## 15. 브라우저 콘솔/네트워크 테스트

- 절차: 주요 화면 방문 후 Playwright console error와 failed network request를 수집한다.
- 통과 기준: 예외, hydration/runtime error, 의도하지 않은 4xx/5xx 요청이 없어야 한다.
- 관련 수정: 프론트엔드 안정성, API 클라이언트 타입 계약 강화.

## 16. 로그 민감정보 테스트

- 절차: 비밀번호, DB URL, API key 형태 문자열을 포함한 mock 오류를 발생시킨다.
- 통과 기준: 서버/브라우저 로그에 원문 secret이 남지 않고 마스킹된다.
- 관련 수정: 민감정보 로그 방지, 로그 수집 라우트 정합성 확인.

## 17. 배포 preflight 테스트

- 절차: build, lint, backend import, API smoke, env check를 하나의 preflight로 순서 실행한다.
- 통과 기준: 성공 시 요약 리포트, 실패 시 실패 단계와 수정 힌트가 출력된다.
- 관련 수정: 배포 전 체크리스트 자동화.

## 18. 성능 기준 테스트

- 절차: list/search/detail/admin stats API의 응답 시간을 샘플링한다.
- 통과 기준: 기준 시간을 초과한 endpoint와 원인을 리포트한다.
- 관련 수정: API 응답 시간 측정 추가, 프론트엔드 체감 성능 개선.

## 현재 확인된 게이트 상태

- `npm run build`: 통과.
- `npm run lint`: 실패, `.vite` 포함 및 source lint error 존재.
- `python -m compileall backend`: 통과.
- `backend.main` import: 실패, 현재 환경에서 `psycopg2` 누락.
