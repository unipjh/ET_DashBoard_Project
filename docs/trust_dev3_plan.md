## 0. 목적 재정의 (핵심 전제)

> trust.py는 **사실 여부(fact-check)를 판별하는 시스템이 아니다.**
> "이 기사가 가짜뉴스 구조적 패턴을 띠는가"를 탐지하여
> 사용자에게 비판적 독해를 유도하는 **패턴 경고 시스템**이다.

이 전제 하에 아래의 가중치, 검증, UX 방안이 설계된다.

---

## 1. 현재 가중치 분석

```python
# 현재 (Phase 2)
WEIGHTS = {
    "source_credibility":  0.25,   # 출처 신뢰성
    "evidence_support":    0.25,   # 근거 지지도
    "style_neutrality":    0.20,   # 문체 중립성
    "logical_consistency": 0.20,   # 논리 일관성
    "clickbait_risk":     -0.10,   # 어뷰징 위험도 (감산)
}
```

### 문제점

| 기준 | 현재 비중 | 가짜뉴스 패턴 기여도 | 판정 |
|------|-----------|---------------------|------|
| source_credibility | 25% | 중간 — 조작 기사도 언론사 이름 달기도 함 | 소폭 하향 여지 |
| evidence_support | 25% | 높음 — 근거 없는 주장은 핵심 패턴 | 유지 |
| style_neutrality | 20% | **매우 높음** — 감정적 선동은 가장 명확한 신호 | **상향 필요** |
| logical_consistency | 20% | 중간 — 조작 기사의 내부 모순 탐지 | 유지 또는 소폭 하향 |
| clickbait_risk | -10% | **매우 높음** — 제목-본문 불일치는 조작 의도 직접 신호 | **페널티 강화 필요** |

## 2. 가중치 실험 설계

### 2-1. 실험 후보군

총 4개 가중치 조합을 테스트한다.

| 조합 | source | evidence | style | logic | clickbait | 설명 |
|------|--------|----------|-------|-------|-----------|------|
| **W0** (현재) | 0.25 | 0.25 | 0.20 | 0.20 | -0.10 | 기준선 |
| **W1** (스타일 강화) | 0.20 | 0.25 | 0.25 | 0.15 | -0.15 | style + clickbait 강화 |
| **W2** (어뷰징 집중) | 0.20 | 0.20 | 0.25 | 0.15 | -0.20 | clickbait 최대 페널티 |
| **W3** (균형 강화) | 0.20 | 0.25 | 0.25 | 0.20 | -0.10 | style만 올리고 나머지 소폭 조정 |

> **참고**: `_MAX_RAW`는 `sum(abs(w)) * 10`으로 동적 계산되므로
> 가중치 합계가 바뀌어도 0~100 스케일은 자동 유지된다.

### 2-2. 실험 실행 방법

각 조합을 `eval_trust.py`에 파라미터로 주입하거나,
`trust.py` 상단의 `WEIGHTS`를 직접 교체 후 실행한다.

```bash
# WEIGHTS를 W1으로 바꾼 후
python -m backend.services.eval_trust --data data/trust_eval_snu.jsonl

# 결과를 조합명과 함께 기록
```

> **비용 절감 팁**: Gemini API 호출이 비싸므로
> LLM 응답(per_criteria 점수)을 캐싱한 후 가중치만 재계산하는 방식을 권장.
> → `eval_trust.py`에 `--cached` 모드 추가 고려 (per_criteria JSON을 파일로 저장)

## 3. 검증 방법론 — 불변 조건 테스트

SNU 팩트체크 데이터는 사실 정확성 레이블이므로 패턴 탐지 목적과 맞지 않는다.
대신 **불변 조건 쌍 테스트(Invariant Pair Test)** 를 사용한다.

### 3-1. 테스트 원리

```
원본 기사 A vs 조작 요소 추가한 기사 A'
→ score(A) > score(A') 가 성립해야 함
→ 절대 점수가 아닌 순위(방향성)만 검증
```

### 3-2. 테스트 쌍 설계 (총 10쌍 목표)

| 쌍 # | 신호 종류 | A (원본) | A' (조작) | 기대 결과 |
|------|-----------|---------|-----------|-----------|
| P1 | 출처 신호 | 연합뉴스 기사 (source 있음) | 동일 텍스트, source=None | score(A) > score(A') |
| P2 | 근거 신호 | 전문가 인용·통계 있는 기사 | 인용/통계 문장 제거 | score(A) > score(A') |
| P3 | 감정 신호 | 중립 문체 기사 | "충격", "경악", "대박" 삽입 | score(A) > score(A') |
| P4 | 제목 신호 | 제목-본문 일치 기사 | 자극적 제목으로 교체 | score(A) > score(A') |
| P5 | 논리 신호 | 논리 일관된 기사 | 결론과 모순되는 문장 삽입 | score(A) > score(A') |
| P6 | 복합 신호 | 연합뉴스 팩트 기사 | 출처불명 + 인용 제거 + 자극적 제목 | score(A) >> score(A') |
| P7 | 출처 등급 | 조선일보 기사 (tier 8) | 동일 텍스트, source="블로그뉴스" | score(A) > score(A') |
| P8 | 근거 강도 | 약한 근거 기사 | 전문가 인용 추가 | score(A') > score(A) |
| P9 | 중립 기사 | 단신 속보 (짧고 근거 없음) | — | score ≈ 40~55 (uncertain 범위) |
| P10 | 명백한 조작 | 루머 기사 (출처불명+선동적) | — | score < 40 (likely_false) |

### 3-3. 판정 기준

| 결과 | 해석 |
|------|------|
| 10쌍 중 ≥8쌍 방향 일치 | 가중치 조합 채택 |
| 10쌍 중 6~7쌍 일치 | 특정 기준 재검토 후 재실험 |
| 10쌍 중 ≤5쌍 일치 | 해당 가중치 조합 기각 |

## 4. 최적화 절차 (단계별)

```
Step 1. 테스트 쌍 10개 수작업 구성 (data/invariant_pairs.jsonl)
         ↓
Step 2. W0(현재) 기준선 테스트 → 몇 쌍 통과하는지 확인
         ↓
Step 3. W1, W2, W3 순서로 실험
         ↓
Step 4. 통과율 가장 높은 조합 채택
         ↓
Step 5. 채택 조합으로 verdict 임계값 재보정
         → 통과한 쌍들의 점수 분포 보고 70/40 조정 여부 결정
         ↓
Step 6. trust.py WEIGHTS 교체 후 기존 분석 기사 재채점 (선택)
```

### Step 5 상세 — 임계값 재보정 방법

```python
# 현재
≥ 70 → "likely_true"
≥ 40 → "uncertain"
<  40 → "likely_false"

# 재보정 방법:
# 1. P9 (단신 속보) 점수가 uncertain 범위(40~69)에 들어오는지 확인
# 2. P10 (루머 기사) 점수가 < 40인지 확인
# 3. 위 두 조건을 만족하는 임계값으로 조정
```

## 5. LLM 응답 캐싱 구현안 (비용 절감)

가중치 실험 시 동일 기사를 매번 Gemini에 보내면 비용이 과다하다.
`per_criteria` 점수를 한 번만 받아서 재활용하는 구조:

```python
# eval_trust.py 확장
# --save-raw  : LLM 응답(per_criteria JSON)을 data/eval_cache.jsonl에 저장
# --from-raw  : 캐시에서 per_criteria 로드 후 가중치만 재계산 (LLM 호출 없음)

python -m backend.services.eval_trust --data data/invariant_pairs.jsonl --save-raw
# → data/eval_cache.jsonl 생성

python -m backend.services.eval_trust --from-raw data/eval_cache.jsonl --weights W1
python -m backend.services.eval_trust --from-raw data/eval_cache.jsonl --weights W2
# → LLM 호출 0회, 가중치만 바꿔가며 즉시 비교 가능
```

## 6. UX 방안

### 6-1. 핵심 원칙

> 점수를 "판결"이 아닌 "체크리스트 결과"로 제시한다.
> 사용자가 스스로 판단할 수 있도록 **근거를 함께 노출**한다.

### 6-2. verdict 표현 변경

| 현재 코드값 | 현재 UI 표현 | 제안 UI 표현 | 의도 |
|------------|-------------|-------------|------|
| `likely_true` | "신뢰 가능" | "패턴 정상" 또는 "주요 경고 없음" | 사실이라는 주장 배제 |
| `uncertain` | "불확실" | "일부 주의 필요" | 특정 기준에서 경고 발생 |
| `likely_false` | "신뢰 불가" | "가짜뉴스 패턴 감지" | 구조적 문제 경고 |

### 6-3. 점수 대신 기준별 경고 카드

숫자 점수(67점)보다 기준별 결과를 카드로 보여주는 방식이 UX적으로 효과적이다.

```
┌──────────────────────────────────────────────────┐
│  ⚠️ 일부 주의 필요                                 │
│                                                  │
│  ✅ 출처 신뢰성    연합뉴스 (공신력 있는 매체)      │
│  ✅ 근거 지지도    전문가 인용 3건, 통계 포함       │
│  ⚠️ 문체 중립성    감정적 표현 일부 감지            │
│  ✅ 논리 일관성    주장과 결론 일치                 │
│  ⚠️ 어뷰징 위험    제목과 본문 내용 일부 불일치     │
│                                                  │
│  이 기사는 구조적으로 일부 주의가 필요합니다.       │
│  내용의 사실 여부는 직접 확인하세요.               │
└──────────────────────────────────────────────────┘
```

### 6-4. 면책 문구 (필수)

모든 신뢰도 결과 하단에 고정 표시:

```
"이 분석은 기사의 작성 패턴을 평가한 것으로,
 사실 여부를 보장하지 않습니다.
 중요한 내용은 원문 출처를 직접 확인하세요."
```

### 6-5. TrustGauge 컴포넌트 개선 방향

현재 `frontend/src/components/TrustGauge.tsx`는 점수 게이지만 표시한다.
아래 정보를 추가하면 UX 가치가 크게 올라간다:

| 추가 요소 | 구현 방법 | 효과 |
|-----------|-----------|------|
| 기준별 아이콘 + 한줄 요약 | `per_criteria.reason` 활용 | "왜 이 점수인지" 전달 |
| 경고 기준 하이라이트 | 점수 < 5인 기준 강조 표시 | 주의 포인트 즉시 인지 |
| 면책 툴팁 | 점수 옆 `?` 아이콘 | 오해 방지 |
| verdict 텍스트 교체 | 위 6-2 표 기준 | 사실 판별 오해 차단 |
