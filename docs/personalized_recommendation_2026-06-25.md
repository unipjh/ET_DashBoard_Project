# 개인화 추천 시스템 구현 문서

> 작성일: 2026-06-25  
> 대상 독자: AI 개념에 익숙한 비전공자  
> 프로젝트: ET by Claude (Korean News Trust Platform)

---

## 1. 목표

사용자가 본 기사 기록을 바탕으로 **"이 사람이 다음에 읽을 법한 기사"** 를 자동으로 추천한다.

단순히 "많이 읽힌 기사" 나 "같은 카테고리 기사"가 아니라, 세션 내 읽기 패턴을 실시간으로 반영한다. 예를 들어 이번 세션에서 IT 기사를 3개 연속으로 읽은 사람에게는 같은 IT 기사를 우선 보여준다.

---

## 2. 전체 추천 플로우

```
[사용자가 메인 페이지 접속]
        │
        ▼
GET /api/recommendations?session_id=xxx
        │
        ▼
┌─────────────────────────────────────┐
│  최근 읽은 기사 이력 조회 (최대 20개)  │
│  event_logs 테이블 기반              │
└─────────────────────────────────────┘
        │
        ├─ 이력 없음 ──────────────────────▶ [4단계] 최신 기사 반환
        │
        ▼ 이력 있음
┌─────────────────────────────────────┐
│  어텐션 인코더 활성화 여부 확인        │
│  (model_weights/attention_encoders.pt│
│   파일 존재 여부)                    │
└─────────────────────────────────────┘
        │
        ├─ 활성화됨 ──▶ [1단계] 어텐션 인코더 추천
        │                      │ 결과 없으면 아래로
        │
        ▼ 미활성화 또는 결과 없음
[2단계] 임베딩 유사도 추천
        │ 결과 없으면 아래로
        ▼
[3단계] 카테고리 기반 추천
        │ 결과 없으면 아래로
        ▼
[4단계] 최신 기사 반환
```

각 단계는 앞 단계가 실패하거나 결과가 없을 때만 실행되는 **폴백(fallback) 체인** 구조다.

---

## 3. 추천 단계별 상세 설명

### 1단계 — 어텐션 인코더 추천 (가장 정교)

이 단계는 학습된 딥러닝 모델이 동작할 때만 활성화된다.

**작동 원리:**

사용자가 최근에 읽은 기사 시퀀스를 입력으로 받아, 그 순서와 맥락을 반영한 **사용자 취향 벡터** 하나를 만든다. 그리고 이 벡터와 가장 유사한 기사를 DB에서 찾아 반환한다.

```
읽은 기사1 ──┐
읽은 기사2 ──┤──▶ UserEncoder ──▶ 사용자 벡터(768차원)
읽은 기사3 ──┘                          │
                                        ▼
                              코사인 유사도 검색
                                        │
                                        ▼
                              추천 기사 top-K 반환
```

단순 평균이 아니라 **어텐션(attention)** 을 사용하기 때문에, 최근에 읽은 기사에 더 큰 가중치를 주고 읽기 시퀀스의 맥락을 포착한다. (예: IT → IT → 경제 순서로 읽었다면, "IT 독자가 경제에도 관심 생김" 패턴을 학습)

---

### 2단계 — 임베딩 유사도 추천

어텐션 인코더가 없거나 결과가 없을 때의 첫 번째 폴백.

읽은 기사들의 **Gemini 임베딩 벡터를 평균** 내어 프로필 벡터를 만들고, 이와 가장 비슷한 기사를 찾는다. 최근 기사일수록 더 높은 가중치를 부여한다.

```
읽은 기사 임베딩들 ──▶ 가중 평균 ──▶ 프로필 벡터
                                          │
                              pgvector 코사인 유사도 검색
                                          │
                                    추천 기사 반환
```

학습 없이 즉시 동작하지만, 어텐션 인코더에 비해 순서 정보(시퀀스)를 반영하지 못한다.

---

### 3단계 — 카테고리 기반 추천

읽은 기사들의 카테고리를 추출해서, 같은 카테고리의 최신 기사를 반환한다. 임베딩이 없는 기사도 커버할 수 있다.

```
읽은 기사들 ──▶ 카테고리 추출 ──▶ 동일 카테고리 최신 기사 반환
                 (예: IT/과학, 경제)
```

---

### 4단계 — 최신 기사 반환 (최후 폴백)

아무 정보도 없거나 모든 단계가 실패할 때, 단순히 가장 최근에 수집된 기사를 반환한다.

---

## 4. 어텐션 인코더 내부 구조

어텐션 인코더는 두 개의 신경망 모듈로 구성된다. NRMS(Neural News Recommendation with Multi-Head Self-Attention) 논문 구조를 따른다.

### NewsEncoder — 기사 하나를 벡터로 변환

기사 하나를 세 가지 시각에서 표현하고, 어텐션으로 합친다.

```
제목 임베딩(768d)    ──┐
요약 임베딩(768d)    ──┼──▶ 어텐션 가중 합산 ──▶ 기사 벡터(768d)
카테고리 원핫 인코딩  ──┘
```

"이 기사에서 제목이 더 중요한가, 요약이 더 중요한가"를 데이터로부터 학습한다.

### UserEncoder — 읽기 시퀀스를 취향 벡터로 변환

최근 읽은 기사 벡터들(최대 20개)을 받아서 사용자 벡터 하나로 압축한다.

```
[기사1벡터, 기사2벡터, ..., 기사N벡터]
        │
        ▼
  Multi-Head Self-Attention
  (기사들 간 상호 관계 포착)
        │
        ▼
  Additive Attention Pooling
  (중요한 기사에 더 집중)
        │
        ▼
  사용자 벡터(768d)
```

Self-Attention이 핵심이다. "이 사람이 IT 기사 읽고 경제 기사 읽었다면, 이 두 기사 사이의 관계가 의미 있다"는 것을 학습한다.

## 5. 로그 수집 — 학습 데이터의 원천

추천 모델을 학습하려면 **"누가 어떤 기사를 봤고, 어떤 기사를 실제로 클릭했는가"** 데이터가 필요하다. 이것을 Supabase `event_logs` 테이블에 자동으로 기록한다.

| 이벤트 타입 | 의미 | 학습에서의 역할 |
|-------------|------|----------------|
| `impression` | 기사 목록이 화면에 표시됨 | **노출됐지만 클릭 안 한 기사** → 학습의 부정 샘플(negative) |
| `click_article` | 기사 클릭 | **클릭한 기사** → 학습의 긍정 샘플(positive) |
| `view_detail` | 기사 상세 페이지 진입 | 긍정 샘플 (강한 신호) |
| `visit_main` | 메인 페이지 방문 | 세션 구분 용도 |

학습 샘플 구조 예시:
```
세션 A의 기록:
  impression: [기사1, 기사2, 기사3, 기사4, 기사5]  ← 5개 노출됨
  click:       기사3                                 ← 1개 클릭

→ 학습 샘플:
   history: [이전에 읽은 기사들]
   positive: 기사3          ← 모델이 높게 점수 매겨야 함
   negatives: [기사1, 기사2, 기사4, 기사5]  ← 모델이 낮게 점수 매겨야 함
```

## 6. 합성 데이터 학습 (현재 적용 상태)

실제 사용자 로그가 충분히 쌓이기 전, **가상의 사용자 패턴을 설계해서 초기 모델을 학습**했다.

### 8개 페르소나 설계

| 페르소나 | 주 관심사 | 보조 관심사 | 특이점 |
|----------|-----------|------------|--------|
| IT마니아 | IT/과학 70% | 경제 15% | — |
| 경제독자 | 경제 70% | 정치 15% | — |
| 정치독자 | 정치 70% | 사회 15% | — |
| 사회독자 | 사회 60% | 생활/문화 20% | — |
| 국제독자 | 세계 70% | 정치 15% | — |
| 라이프독자 | 생활/문화 60% | 연예 25% | — |
| 종합독자 | 전 카테고리 균등 | — | — |
| 신뢰독자 | 전 카테고리 균등 | — | trust_score 높은 기사 선호 |

각 페르소나는 100번의 세션을 시뮬레이션한다.  
매 세션마다 15개 기사를 노출받고, 카테고리 선호도 + 약간의 랜덤 노이즈로 1개를 "클릭"한다.

```
페르소나 "IT마니아" 세션 시뮬레이션:
  노출 풀: [IT기사A, 경제기사B, 사회기사C, IT기사D, ...]
  점수:   [0.73,    0.18,      0.05,      0.68, ...]
               ↑ 노이즈 포함
  클릭: IT기사A (최고 점수)
  
  → InteractionSample(
       history=[이전 클릭들],
       positive=IT기사A,
       negatives=[경제기사B, 사회기사C, ...]
    )
```

총 **792개 샘플** 생성 후 학습 (8 페르소나 × 99세션).

### 학습 결과

```
epoch=1  loss=1.51  val_auc=0.500
epoch=2  loss=1.48  val_auc=0.528
epoch=3  loss=1.46  val_auc=0.536
epoch=4  loss=1.45  val_auc=0.519
epoch=5  loss=1.44  val_auc=0.524
```

- **val_auc 0.53**: 랜덤(0.50) 대비 카테고리 패턴을 학습함
- **baseline_auc 0.00**: DB 기사들에 아직 Gemini 임베딩(`embed_summary`)이 없어서 임베딩 베이스라인이 0벡터 → 비교 불가 상태
- 임베딩이 채워지면 성능이 크게 향상될 것으로 예상

> **왜 합성 데이터인가?**  
> 실제 사용자 로그는 최소 100회 이상의 클릭 + 1,000회 이상의 impression이 쌓여야 의미 있는 학습이 가능하다. 초기 배포 직후에는 데이터가 없으므로, 의미 있게 설계된 합성 데이터로 모델을 부트스트랩한다. 실제 데이터가 충분히 쌓이면 재학습으로 교체한다.

## 7. 전체 시스템 파일 구조

```
backend/
├── routers/
│   └── recommendations.py       ← GET /api/recommendations 엔드포인트
├── services/
│   ├── recommend.py             ← 추천 로직 + 폴백 체인
│   └── encoder_inference.py     ← 학습된 모델 로드 및 추론
└── training/
    ├── news_encoder.py          ← NewsEncoder 모델 정의
    ├── user_encoder.py          ← UserEncoder 모델 정의
    ├── dataset.py               ← 학습 데이터 로딩 (DB / 합성)
    ├── generate_synthetic_samples.py  ← 페르소나 기반 합성 샘플 생성
    ├── train_user_encoder.py    ← 학습 실행 스크립트
    ├── export_weights.py        ← 체크포인트 → 서빙 경로 복사
    ├── backfill_learned_embeddings.py ← 기존 기사 learned_embedding 채우기
    └── checkpoints/
        └── attention_encoders.pt ← 학습된 가중치

frontend/
└── src/api/logger.ts            ← impression/click 이벤트 자동 전송
```

## 8. 재학습 가이드

### 실제 데이터 충분히 쌓인 후 재학습

아래 조건이 모두 충족되면 실제 데이터로 재학습한다:

```sql
-- Supabase에서 확인
SELECT event_type, count(*)
FROM event_logs
WHERE event_type IN ('click_article', 'view_detail', 'impression')
GROUP BY event_type;
```

| 조건 | 기준 |
|------|------|
| 클릭/뷰 이벤트 합계 | ≥ 100회 |
| impression 이벤트 | ≥ 1,000회 |
| 경과 기간 | 2~3일 이상 |

```powershell
# 실제 데이터로 재학습
python -m backend.training.train_user_encoder --epochs 10

# 합성 데이터 + 실제 데이터 혼합 (아직 데이터가 적을 때)
# → 현재는 아직 지원 안 함, 나중에 --blend-synthetic 플래그 추가 예정
```

### 합성 데이터로만 재학습 (현재 방식)

```powershell
# 기본 실행 (페르소나당 100세션)
python -m backend.training.train_user_encoder --synthetic --epochs 5

# 더 많은 데이터로 (페르소나당 200세션)
python -m backend.training.train_user_encoder --synthetic --epochs 10 --synthetic-sessions 200

# 가중치 export (학습 후 반드시 실행)
python -m backend.training.export_weights

# 기존 기사에 learned_embedding 채우기
python -m backend.training.backfill_learned_embeddings
```

## 9. 현재 상태 및 다음 단계

### 현재 (2026-06-25)

| 컴포넌트 | 상태 |
|----------|------|
| `GET /api/recommendations` 엔드포인트 | ✅ 운영 중 |
| impression/click 로그 수집 | ✅ 운영 중 |
| 어텐션 인코더 (합성 데이터 학습) | ✅ 배포됨 |
| 실제 사용자 데이터 기반 학습 | ⏳ 대기 중 |
| Gemini 임베딩 채워진 기사 | ⏳ 크롤+분석 실행 필요 |

### 다음 할 일

1. **크롤 + 분석 실행** — 어드민 페이지에서 기사 수집 후 신뢰도 분석을 실행하면 `embed_summary`(Gemini 임베딩)가 채워진다. 임베딩이 있어야 추천의 2단계(임베딩 유사도)와 어텐션 인코더 성능이 제대로 발휘된다.

2. **데이터 축적 확인** — 실제 사용자 로그가 위 기준에 도달하면 `train_user_encoder.py`를 (--synthetic 없이) 재실행한다.

3. **성능 모니터링** — 재학습 후 val_auc가 합성 데이터 학습 결과(0.53)보다 높으면 실제 데이터 기반 모델이 더 잘 동작하는 것으로 판단한다.
