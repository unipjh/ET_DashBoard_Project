# 수정계획_2026-05-31

## 기준

- 대상: ET Dashboard / Explainable Trust 현재 프로젝트
- 작성 기준: 문제 확인 > 수정 제안 > 테스트 제안
- 우선순위: P0는 즉시 안정화, P1은 핵심 기능 신뢰성, P2는 실사용 품질, P3는 개선 여지

## P0. 저장소와 배포 안정성

### 1. 로컬과 GitHub 기준선 정리

- 문제 확인: 현재 `clean-main`은 `origin/main`보다 6커밋 뒤처져 있고 로컬 수정도 존재한다.
- 수정 제안: 원격 변경, 로컬 변경, 충돌 예상 파일을 분리한 뒤 병합 순서를 확정한다.
- 테스트 제안: 병합 전후 `git diff --stat`, `npm run build`, 백엔드 import 확인을 통과시킨다.

### 2. 중첩 Git 저장소 처리

- 문제 확인: `ET_DashBoard_Project` 하위 repo는 tracked 파일이 모두 삭제 상태로 잡힌다.
- 수정 제안: 이 폴더를 복구 대상, 삭제 대상, ignore 대상 중 하나로 명확히 결정한다.
- 테스트 제안: 결정 후 루트와 하위 repo 각각 `git status --short --branch`가 의도대로 보이는지 확인한다.

### 3. 백엔드 DB 의존성 일관화

- 문제 확인: `backend.main` import가 `psycopg2` 누락으로 실패한다.
- 수정 제안: 루트와 `backend/requirements.txt`의 PostgreSQL 관련 의존성을 동일하게 맞춘다.
- 테스트 제안: 새 환경 기준으로 `python -c "from backend.main import app"`가 성공해야 한다.

### 4. generated 파일 추적 제거

- 문제 확인: `frontend/.vite` 캐시 파일이 Git과 ESLint 대상에 포함되어 린트를 오염시킨다.
- 수정 제안: `.vite` 추적을 제거하고 ESLint ignore에도 `dist`, `.vite`, generated 산출물을 명시한다.
- 테스트 제안: `git ls-files frontend/.vite`가 비고, `npm run lint`가 소스 파일만 검사해야 한다.

## P1. 기술 안정성

### 5. 백엔드 smoke test 추가

- 문제 확인: API가 import, health, admin auth를 자동 검증하는 테스트가 없다.
- 수정 제안: `pytest` 기반으로 app import, `/`, `/health`, admin 401 테스트를 추가한다.
- 테스트 제안: DB 실패 mock에서도 `/health`가 503을 반환하는지 확인한다.

### 6. 관리자 작업 상태 신뢰성 개선

- 문제 확인: crawl/analyze/dedupe 상태가 프로세스 메모리 중심이라 재시작과 동시 실행에 취약하다.
- 수정 제안: 작업 ID, 상태, 시작/종료시각, 에러 메시지를 명시하는 상태 모델을 만든다.
- 테스트 제안: 작업 시작, 진행, 실패, 완료 응답을 fixture로 검증한다.

### 7. 외부 API 실패 처리 표준화

- 문제 확인: Gemini, Naver, DB 오류가 사용자에게 동일한 품질로 전달되지 않는다.
- 수정 제안: 외부 연동 오류를 retryable, fatal, partial success로 분류해 응답과 로그를 통일한다.
- 테스트 제안: timeout, rate limit, empty response mock으로 관리자 화면 메시지를 확인한다.

### 8. CORS와 환경변수 검증

- 문제 확인: 필수 환경변수 누락 시 런타임 실패 시점이 늦고 원인 파악이 어렵다.
- 수정 제안: startup 시 `DATABASE_URL`, `GEMINI_API_KEY`, `ADMIN_PASSWORD`, `ALLOWED_ORIGINS`를 검증한다.
- 테스트 제안: env 누락 케이스별로 명확한 startup 경고 또는 실패 메시지를 확인한다.

## P1. 기술 품질

### 9. 프론트엔드 lint baseline 복구

- 문제 확인: `any`, 빈 catch, 미사용 변수, hook rule 오류로 `npm run lint`가 실패한다.
- 수정 제안: API 타입을 중심으로 공용 타입을 정리하고 빈 catch는 명시적 fallback으로 바꾼다.
- 테스트 제안: `npm run lint`가 0 error로 종료되어야 한다.

### 10. API 클라이언트 타입 계약 강화

- 문제 확인: 화면이 API 응답 필드를 암묵적으로 기대하고 일부는 `any`로 우회한다.
- 수정 제안: article, stats, feedback, process status 타입을 하나의 계약으로 정리한다.
- 테스트 제안: mock 응답으로 list/detail/admin/feedback 렌더가 타입 에러 없이 통과해야 한다.

### 11. 백엔드 repository 계층 분리

- 문제 확인: `repo.py` 변경량이 크고 DB, 검색, 변환 책임이 한 파일에 집중되어 있다.
- 수정 제안: connection, article query, search, feedback, admin stats 단위로 책임을 분리한다.
- 테스트 제안: 분리 전후 동일 fixture에서 동일 API 응답을 반환해야 한다.

### 12. 로그 수집 라우트 정합성 확인

- 문제 확인: GitHub 원격에는 logs 라우트와 logger가 추가되어 있으나 로컬과 차이가 있다.
- 수정 제안: 원격 로그 기능을 병합할지 결정하고 프론트/백엔드 로그 계약을 확정한다.
- 테스트 제안: 브라우저 오류 발생 시 서버 로그 endpoint에 민감정보 없이 기록되는지 확인한다.

## P1. UI/UX

### 13. 관리자 인증 UX 정리

- 문제 확인: 인증 실패, 세션 만료, 비밀번호 변경 흐름이 관리자 작업 중단과 섞인다.
- 수정 제안: 로그인, 실패, 재인증, 로그아웃 상태를 분리하고 명확한 메시지를 제공한다.
- 테스트 제안: 잘못된 비밀번호, 만료 후 재시도, 새로고침 복구 흐름을 Playwright로 확인한다.

### 14. 관리자 작업 실행 UX 개선

- 문제 확인: crawl 시작 범위, 진행률, 완료 결과, 실패 원인이 충분히 예측 가능하지 않다.
- 수정 제안: 실행 전 범위 요약, 실행 중 단계 표시, 완료 후 추가/중복/실패 수를 표시한다.
- 테스트 제안: 성공, 중복 0건, 서버 오류, 네트워크 오류 상태가 각각 다른 메시지로 보여야 한다.

### 15. 빈 상태와 오류 상태 보강

- 문제 확인: 기사 없음, 검색 결과 없음, DB 오류, 피드백 없음 상태가 기능 안내로 연결되지 않는다.
- 수정 제안: 각 빈/오류 상태에 재시도, 필터 초기화, 관리자 이동 등 다음 행동을 제공한다.
- 테스트 제안: API empty/error mock에서 화면이 깨지지 않고 CTA가 노출되는지 확인한다.

### 16. 모바일 레이아웃 검증

- 문제 확인: 테이블, 관리자 카드, 긴 기사 제목, trust 설명이 모바일에서 overflow 위험이 있다.
- 수정 제안: 모바일에서는 표를 카드형 목록으로 전환하고 긴 텍스트 줄바꿈 규칙을 고정한다.
- 테스트 제안: 375px, 768px, 1440px 뷰포트에서 텍스트 겹침이 없어야 한다.

### 17. 신뢰도 설명 가독성 개선

- 문제 확인: trust score와 criteria가 사용자가 판단할 수 있는 설명 구조로 충분히 정리되지 않았다.
- 수정 제안: 점수, 판정, 근거, 기준별 사유, 원문 근거를 구분해 보여준다.
- 테스트 제안: trust 데이터 누락, 0점, 긴 사유, JSON 파싱 실패 케이스를 렌더 검증한다.

## P1. 검색과 데이터 품질

### 18. 검색 결과 신뢰성 기준 추가

- 문제 확인: hybrid search 결과가 관련도, 중복, 빈 결과 측면에서 자동 검증되지 않는다.
- 수정 제안: 고정 fixture로 semantic, keyword, hybrid 결과 순위를 비교하는 테스트를 만든다.
- 테스트 제안: 동일 query에서 top result와 중복 제거 결과가 기대값과 일치해야 한다.

### 19. 기사 중복 제거 기준 명확화

- 문제 확인: dedupe 기준이 title 유사도 중심이라 오삭제와 미삭제를 구분하기 어렵다.
- 수정 제안: 제목, URL, 출처, 발행일을 함께 쓰는 중복 판정 요구사항을 정의한다.
- 테스트 제안: 동일 기사, 제목만 유사한 기사, 같은 제목 다른 출처 케이스를 fixture로 검증한다.

### 20. 키워드 추출 데이터 품질 개선

- 문제 확인: keywords 누락/형식 오류가 검색과 관리자 통계 품질에 직접 영향을 준다.
- 수정 제안: keywords는 배열 형태로 정규화하고 최소/최대 개수와 금지값을 검증한다.
- 테스트 제안: 빈 값, 문자열 JSON, 잘못된 배열, 과도한 키워드 수를 자동 보정 또는 실패 처리한다.

## P2. 보안과 운영

### 21. 관리자 비밀번호 저장 방식 개선

- 문제 확인: 관리자 비밀번호가 sessionStorage에 저장되어 브라우저 노출면이 있다.
- 수정 제안: 단기 토큰 또는 서버 검증 세션으로 바꾸고 저장 시간을 제한한다.
- 테스트 제안: 세션 만료, 로그아웃, 새 탭, 새로고침에서 접근 제어가 유지되는지 확인한다.

### 22. 민감정보 로그 방지

- 문제 확인: API 오류와 관리자 작업 로그가 비밀번호, DB URL, 본문 데이터를 노출할 가능성이 있다.
- 수정 제안: 로그 redaction 규칙을 만들고 secret-like 문자열을 마스킹한다.
- 테스트 제안: 의도적으로 secret 문자열을 포함한 오류를 발생시켜 로그에 원문이 남지 않는지 확인한다.

### 23. 배포 전 체크리스트 자동화

- 문제 확인: Vercel/Render 배포 전 수동 확인 항목이 문서와 실제 게이트로 분리되어 있다.
- 수정 제안: build, lint, backend import, env check, smoke test를 하나의 preflight로 묶는다.
- 테스트 제안: preflight 명령 하나로 성공/실패 요약이 출력되어야 한다.

## P2. 관찰성과 성능

### 24. API 응답 시간 측정 추가

- 문제 확인: list/search/detail/admin API의 체감 속도 기준이 없다.
- 수정 제안: endpoint별 응답 시간과 실패율을 로그 또는 lightweight metric으로 남긴다.
- 테스트 제안: 95th percentile 목표치를 정하고 mock 부하에서 기준 초과를 감지한다.

### 25. 프론트엔드 체감 성능 개선

- 문제 확인: 기사 목록, 상세, 관리자 화면의 loading skeleton과 stale 데이터 정책이 일관되지 않다.
- 수정 제안: React Query staleTime, retry, loading skeleton, optimistic refresh 정책을 화면별로 통일한다.
- 테스트 제안: 느린 API mock에서 레이아웃 shift 없이 loading과 data 전환이 보여야 한다.
